## Design System

| Property | Value | Source |
|----------|-------|--------|
| Tool | none (shadcn gate skipped — Angular 21 stack) | stack constraint |
| Preset | not applicable | — |
| Component library | PrimeNG 21.1.x (Aura preset) | `app.config.ts` → `providePrimeNG({ theme: { preset: Aura } })` |
| Styling | Tailwind CSS v4 + `tailwindcss-primeui` | `frontend/src/styles.css` |
| Icon library | PrimeIcons 7.x (imported via `primeicons/primeicons.css`) | `frontend/src/styles.css` |
| Font | Aura default stack (system-ui / Inter fallback) — no custom font family declared | `@primeng/themes/aura` |
| Locale | `fr-FR` (registered in `app.config.ts`) — all copy in French | D-global, Phase 2/3/4/5 |
| Currency formatting | `Intl.NumberFormat('fr-FR', { style: 'currency', currency: 'EUR' })` | `transactions.ts#formatAmount` |
| Theme mode | Aura light + dark auto (PrimeNG token-driven). Contract uses `surface-*`, `text-muted-color`, `primary` semantic tokens — never raw hex. | Phase 3/4/5 convention |

**Design system principle:** every color, border, radius, and state comes from PrimeNG Aura tokens or Tailwind utilities mapped to them via `tailwindcss-primeui`. No new CSS variables. No new custom colors. Status badges reuse existing `p-tag` severities (`success` / `warn` / `danger` / `info` / `secondary`) — already theme-aware in both light and dark.

## Color

Palette derived from PrimeNG Aura tokens. No raw hex in code — reference semantic tokens only.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-surface-0` (light) / `bg-surface-950` (dark) | Page background, table rows, dialog body |
| Secondary (30%) | `bg-surface-50` / `bg-surface-100` + `border-surface-200` | Filter bar background (`bg-surface-50 rounded-lg p-4`), sidebar, striped rows, progressbar track |
| Accent (10%) | `primary` token (Aura default = indigo family) | Primary CTAs, active sidebar state, progressbar fill when `GREEN`, links inside cells |
| Destructive | `danger` severity | Delete button, "Dépassé" status badge, destructive confirmation accept button |
| Warning | `warn` severity | "Attention" status badge (80-100%), archived-account message |
| Success | `success` severity | "Sur la bonne voie" status badge (<80%), restored-archive toast |
| Info | `info` severity | Rollover indicator pill, scope=SHARED badge, read-only scope field |
| Muted | `text-muted-color` | Secondary text (empty description `—`, category counts, hint copy) |

**Accent reserved for (exhaustive list — never elsewhere):**
1. Primary CTA button fill: "Nouvelle enveloppe" on list page, "Enregistrer" in dialog footer, "Personnaliser ce mois" trigger.
2. Active sidebar link background + left border (pattern from `sidebar.ts`: `bg-surface-100 text-primary font-semibold border-l-3 border-primary`).
3. Pointed/active icon states inside the envelope detail page header (e.g. "Modifier" primary button).
4. `p-progressbar` fill color when envelope status is `GREEN` (<80%). Warn and danger severities override this for YELLOW/RED respectively.
5. `pi-wallet`-class icon when sidebar entry is active.

Accent is NEVER used for: hover states (use `bg-surface-50`), row stripes, filter bar background, category chips, borders, text on body copy, tag backgrounds for non-primary statuses.

### Envelope Status Color Mapping (ENVL-05 / D-13)

Derived server-side into enum `EnvelopeStatus` (GREEN/YELLOW/RED) + numeric `ratio` (0.0..n.nn) in `EnvelopeResponse`. Frontend never re-computes thresholds — it maps status → severity 1:1.

| Ratio | `EnvelopeStatus` | `p-tag` severity | `p-tag` label (French) | `p-progressbar` color | ARIA label template |
|-------|------------------|------------------|------------------------|-----------------------|---------------------|
| `ratio < 0.80` | `GREEN` | `success` | `Sur la bonne voie` | success token (theme green) | `"Enveloppe {name}: sur la bonne voie, {percent}% consommé"` |
| `0.80 ≤ ratio ≤ 1.00` | `YELLOW` | `warn` | `Attention` | warn token (theme amber) | `"Enveloppe {name}: attention, {percent}% consommé"` |
| `ratio > 1.00` | `RED` | `danger` | `Dépassé` | danger token (theme red) | `"Enveloppe {name}: dépassée, {percent}% consommé"` |

Rendering rules:
- `p-progressbar` clamped at 100% display even when `ratio > 1`. Actual percentage shown as the tag value and announced via ARIA label.
- `p-tag` uses the neutral `severity` attribute so both Aura light and Aura dark render AA-compliant contrast automatically — we do not hand-pick hex.
- Display percentage: `Math.round(ratio * 100)` + `%`. Never show decimals.
- When `consumed = 0` and `available = 0` → status `GREEN`, label `Sur la bonne voie`, progressbar at 0%.

### Extended CategorySelector Contract

Input: `selectionMode = 'single' | 'checkbox'` (default `'single'` — existing behaviour unchanged).
Input: `selectedIds` signal/input of `string[]` when in checkbox mode (ignored in single mode).
Output: `categoriesSelected: output<string[]>` emitted in checkbox mode; existing `categorySelected: output<string \| null>` kept in single mode.
Visual: when `selectionMode='checkbox'`, p-treeSelect renders checkbox nodes, `display="chip"` to render selected items as chips with `showClear`. Placeholder = `"Catégories couvertes"`.
Validation: at least one category required in envelope-dialog → disable Save button when empty (same pattern as `transaction-dialog.ts#isValid`).

### Decision: p-treeSelect multi vs p-multiSelect (D-21 discretion)

**Chosen: `p-treeSelect` with `selectionMode="checkbox"`.** Rationale:
- Reuses the exact component users already know from Phase 5 transactions filter (`transactions.ts` line 82) — consistency > novelty.
- Preserves the 2-level hierarchy visually — p-multiSelect flattens parents + children into a single list which hurts legibility.
- Extends the shared `CategorySelector` without introducing a second shared component.
- Hierarchy-aware selection: ticking a parent ticks all children automatically (PrimeNG native checkbox propagation) — aligns with D-02 "lier une catégorie racine embrasse ses sous-catégories".

### Dialog 1 — Create / Edit Envelope (`envelope-dialog.ts`)

Dialog shell identical to `transaction-dialog.ts` (`[modal]="true" [closable]="true" [draggable]="false" [style]="{ width: '36rem' }"`). Header: `"Nouvelle enveloppe"` or `"Modifier l'enveloppe"` (computed).

**Workflow decision (previously open UX question):**
- When opened with `?accountId=<uuid>` present on the list page, the Compte field is **pre-selected and read-only**. The scope badge is visible immediately.
- When opened without account context (rare — only if planner adds a global "+" shortcut), Compte is editable until first save. Scope badge appears dynamically once an account is chosen.

**Fields (vertical stack, `gap-4 p-6`):**

1. **Compte** (required, locked-once-saved)
   - `<p-select>` floating label `"Compte"`, options = accounts accessible with WRITE, disabled in edit mode.
   - Below: `<p-tag [value]="scopeLabel(selectedAccount?.accountType)" [severity]="scopeSeverity(selectedAccount?.accountType)" />` where `scopeLabel('SHARED')='Commun'`, `scopeLabel('PERSONAL')='Personnel'`. Severity: SHARED=`info`, PERSONAL=`secondary`. Tooltip on SHARED: `"Cette enveloppe sera partagée avec tous les utilisateurs ayant accès au compte commun."`
2. **Nom** (required, max 100 chars)
   - `<p-floatlabel variant="on"><input pInputText [(ngModel)]="name" id="name" maxlength="100" class="w-full" /><label for="name">Nom de l'enveloppe</label></p-floatlabel>`
3. **Catégories couvertes** (required, min 1)
   - `<app-category-selector selectionMode="checkbox" [options]="categoryOptions" [selectedIds]="selectedCategoryIds" (categoriesSelected)="onCategoriesSelected($event)" placeholder="Catégories couvertes" />`
   - Helper text below in `text-sm text-muted-color`: `"Une catégorie racine inclut automatiquement ses sous-catégories."`
4. **Budget mensuel par défaut** (required, >= 0)
   - `<p-floatlabel variant="on"><p-inputnumber [(ngModel)]="budget" mode="currency" currency="EUR" locale="fr-FR" [minFractionDigits]="2" [min]="0" inputId="budget" /><label for="budget">Budget mensuel (EUR)</label></p-floatlabel>`
5. **Report (rollover)** (required, default `RESET`)
   - `<p-selectbutton [options]="rolloverOptions" [(ngModel)]="rolloverPolicy" optionLabel="label" optionValue="value" aria-label="Politique de report" />` with options:
     - `{ value: 'RESET', label: 'Remise à zéro' }`
     - `{ value: 'CARRY_OVER', label: 'Report du solde' }`
   - Helper text below: `"Remise à zéro : chaque mois repart du budget par défaut. Report du solde : le reste du mois précédent s'ajoute (ou se soustrait) au budget du mois suivant."`

**Validation (inline, checker-friendly):**
- Save disabled when: `name` blank OR `selectedCategoryIds.length === 0` OR `budget === null` OR `budget < 0` OR `!selectedAccount`.
- Submit error mapping:
  - 403 → `<p-message severity="error" text="Vous n'avez pas les droits pour modifier les enveloppes de ce compte." />`
  - 409 (D-01 duplicate category-per-account) → `<p-message severity="error" text="Une catégorie sélectionnée appartient déjà à une autre enveloppe de ce compte. Choisissez des catégories libres." />`
  - Other → `<p-message severity="error" text="Impossible d'enregistrer l'enveloppe. Veuillez réessayer." />`

**Footer:**
- `<p-button label="Annuler" [text]="true" severity="secondary" (onClick)="cancelled.emit()" />`
- `<p-button [label]="submitting() ? 'Enregistrement...' : 'Enregistrer'" [loading]="submitting()" [disabled]="!isValid || submitting()" (onClick)="save()" />`

### Dialog 2 — Monthly Override ("Personnaliser ce mois") (`envelope-allocation-dialog.ts`)

Triggered by the list-page action button, the details-page header button, and the overrides table row. Minimal surface.

**Header:** `"Personnaliser le budget d'un mois"` (subtitle: envelope name).

**Fields (vertical stack):**
1. **Mois concerné** (required)

## Responsive & Accessibility

**Breakpoints** (Tailwind default):
- `<md` (<768px): filter grid collapses to 1 column. Table horizontal scroll allowed (p-table native overflow wrapper). Status column badge + progressbar stack vertically inside cell. Actions column shows only icon buttons (already the case).
- `md..lg` (768-1280): filter 2 columns. Table all columns visible.
- `>=lg` (1280+): default spacing. Dialog `width: 36rem` caps nicely — never full-screen.

Target support: laptop 1440 → 4K. No mobile-first UX rework — envelope management is a desktop-primary use case (matches Phases 3/4/5).

**Accessibility (WCAG AA minimum):**
- All page containers use `<h1>` once, `<h2>` for sub-sections. No skipped heading levels.
- Filter bar wrapped in `<div role="search" aria-label="Filtrer les enveloppes">` (matches `transactions.ts` line 64).
- Empty state containers use `role="status"` for screen reader announcements.
- Every icon-only button has `pTooltip` + `[attr.aria-label]` with a verb-first French label. Never rely on icon alone.
- `p-progressbar` uses `[attr.aria-label]` with the ARIA label template from the status table above — progressbar value alone is insufficient because the value is clamped at 100.
- Status `p-tag` announces label via default content — `severity` is purely visual. Do not rely on color alone (label + icon communicate status).
- Focus order in envelope-dialog: Compte → Nom → Catégories → Budget → Rollover → Annuler → Enregistrer. Autofocus on Nom when creating (use `afterNextRender` per Phase 2 convention); autofocus on Budget when editing (since Nom is usually unchanged).
- Keyboard: dialog closes on Esc (PrimeNG default). Save on Enter in any field when form is valid (standard p-dialog behavior — do not override).
- ConfirmDialog destructive actions always exposed with `acceptButtonProps.severity='danger'` (match pattern of `transactions.ts#confirmDelete`).
- Color contrast: Aura preset ensures p-tag + p-progressbar severity tokens meet AA in both light and dark. No manual overrides allowed.

## Design Decisions Log (this contract answers open UX questions)

| Question (from prompt) | Decision | Rationale |
|------------------------|----------|-----------|
| Status computed backend vs frontend | **Backend** — `EnvelopeStatus` enum + `ratio` number included in `EnvelopeResponse`. Frontend maps 1:1 to severity + label. | Avoids dashboard/list divergence in Phase 10 and keeps thresholds single-sourced (D-13 still fixed). Aligns with RESEARCH.md §Architecture Patterns. |
| Create workflow — choose account first vs prefilled | **Prefilled** when `?accountId` is on the list URL; editable otherwise. Scope badge is visible as soon as an account is chosen. | Matches existing envelope entry-point (from account context) and still allows a global "New envelope" without breaking. |
| Rollover visible on list — column or icon | **Inline info p-tag** ("Report") appended to Statut cell when `rolloverPolicy === 'CARRY_OVER'`. No dedicated column. | Saves horizontal real estate; keeps the envelope's "mood" (status + rollover) grouped. |

| Archived envelopes visibility | `Afficher les archivées` toggle on list page, default OFF. Archived rows render with `text-muted-color` opacity. Edit disabled (only Unarchive via future Phase 8). | Mirrors Phase 3 account archive toggle behaviour. |
| p-multiSelect vs p-treeSelect | **p-treeSelect `selectionMode="checkbox"`** via extended shared `CategorySelector`. | Consistency with Phase 5 transactions filter, preserves 2-level hierarchy, parent-ticks-children natively. |
