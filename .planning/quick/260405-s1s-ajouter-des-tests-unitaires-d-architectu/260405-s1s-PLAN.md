Extend the existing ArchitectureTest.java (keep all 3 existing rules intact) with the following new ArchUnit rules:

**Layered dependency rules:**

1. `controllers_should_not_depend_on_repositories` — Classes named `*Controller` should not depend on classes named `*Repository`. Controllers must go through Services. Use `noClasses().that().haveSimpleNameEndingWith("Controller").should().dependOnClassesThat().haveSimpleNameEndingWith("Repository")`.

2. `repositories_should_not_depend_on_controllers_or_services` — Classes named `*Repository` should not depend on `*Controller` or `*Service`. Repositories are the bottom layer.

3. `feature_packages_should_not_depend_on_each_other` — Each feature package (auth, account, transaction, category, envelope) should not import from other feature packages. Only shared and banking are allowed as cross-cutting dependencies. Use `noClasses().that().resideInAPackage("com.prosperity.account..")` chained for each feature forbidding imports from the others. Group these as separate rules per feature package OR use a slices approach. The simplest approach: one rule per feature forbidding deps on other features. However, note that `auth` may be needed by `account` (for User entity references). Check imports first — if account imports auth classes (e.g., User), allow account -> auth dependency but forbid the reverse. Actually, use `noClasses().that().resideInAPackage("com.prosperity.transaction..").should().dependOnClassesThat().resideInAnyPackage("com.prosperity.account..", "com.prosperity.category..", "com.prosperity.envelope..", "com.prosperity.auth..")` but ONLY if this passes. If there are legitimate cross-feature deps (like transaction referencing account/category), soften the rule to only check the pairs that should NOT depend on each other, or skip this rule and rely on the existing cycle-free rule.

**IMPORTANT:** Before writing each rule, mentally verify it passes against the existing codebase. For example, `AccountAccess` references `User` from auth package — so account depends on auth. `Transaction` likely references `Account` and `Category`. These are legitimate domain dependencies. Do NOT write rules that will fail against the current codebase. Instead, focus on rules that enforce direction (no reverse deps):
- auth should NOT depend on account, transaction, category, envelope
- shared should NOT depend on any feature (already exists)
- banking should NOT depend on any feature

4. `banking_should_not_depend_on_feature_packages` — Classes in `com.prosperity.banking` should not depend on auth, account, transaction, category, or envelope packages.

**Annotation placement rules:**

5. `controllers_should_be_annotated_with_rest_controller` — Classes named `*Controller` should be annotated with `@RestController`. Import `org.springframework.web.bind.annotation.RestController`.

6. `services_should_be_annotated_with_service` — Classes named `*Service` should be annotated with `@Service`. Import `org.springframework.stereotype.Service`. Note: `CustomUserDetailsService` is annotated with `@Service` so this should pass.

7. `repositories_should_be_interfaces` — Classes named `*Repository` should be interfaces (Spring Data repositories are interfaces).

**Import the necessary ArchUnit and Spring annotation classes.** Use static imports from `ArchRuleDefinition.classes` and `ArchRuleDefinition.noClasses` (already imported).

Keep the existing `@ArchTest` annotation pattern and `static final ArchRule` field style. Follow Google Java Format (the project uses Spotless with google-java-format).
