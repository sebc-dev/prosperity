# Effets de bord security-critical : commit-inside-service avant `raise`

Par défaut, les services sont **transaction-agnostic** : `backend.shared.db.get_db` est le seul propriétaire de la frontière transactionnelle (commit on success, rollback on exception). Les routes ne commitent jamais ; les services flush au besoin et laissent la dépendance gérer le cycle. Ce contrat échoue dans un cas précis : quand l'effet de bord et l'exception sont **indivisibles pour la sécurité**. Exemple matérialisé : `service/refresh_tokens.py:rotate()` détecte un replay de refresh token → doit invalider toute la famille → doit lever `RevokedRefreshTokenError`. Si la famille n'est invalidée que par un `flush()`, le `rollback()` de `get_db` (déclenché par l'exception qui devient le 401) **efface l'invalidation** ; le client reçoit son 401 mais le mécanisme de détection de replay est neutralisé en prod (cf. issue #58, vérifié empiriquement). Sous true race Postgres REPEATABLE READ, le perdant prend `SerializationFailure` (SQLSTATE 40001) → la transaction est abortée par Postgres → toute écriture ultérieure exige de toute façon un nouveau cycle commit (cf. issue #57). Nous décidons que `rotate()` (et toute future opération de la même classe) **commit explicitement avant `raise`** lorsque l'effet de bord matérialise une décision de sécurité que le client ne doit pas pouvoir défaire en provoquant l'exception.

## Considered Options

- **Savepoint nesté (`begin_nested`)** : un savepoint released est mergé dans la transaction parente — si `get_db` rollback la parente, le savepoint disparaît avec elle. Ne survit pas au rollback, **ne résout pas le problème**.
- **Route ouvre une 2e session sur l'exception** : la route catche `RevokedRefreshTokenError`, ouvre une nouvelle session via `app.state.sessionmaker`, fait l'invalidation, commit, raise 401. Couple la route au sessionmaker (contournement de `get_db`) et **scinde une opération logique en deux couches** (service détecte, route écrit). Sous SerializationFailure, la même séquence rollback → nouvelle txn → commit est nécessaire de toute façon ; déplacer ça dans la route ne supprime pas le smell, le déguise.
- **Commit-inside-service (retenu)** : `rotate()` ouvre, écrit et commit explicitement la famille-invalidation avant le `raise`. Le `get_db` qui suit verra une session sans dirty state — son `rollback()` sur exception devient un no-op, le commit déjà fait persiste. Coût : le service connaît la frontière transactionnelle dans **ce path précis**, et le docstring porte la dette.

## Consequences

- **Le contrat de `rotate()` est resserré** : au moment où `RevokedRefreshTokenError` est levée par la branche replay, la famille est garantie **invalidée et persistée**, indépendamment du sort de la transaction parente. Idem quand la branche est atteinte via `SerializationFailure` recovery.
- **Cette dérogation est l'exception, pas la règle.** Les services métier ordinaires (transactions, dettes, settlements, sync handlers) restent transaction-agnostic — c'est `get_db` (ou le write upload handler, cf. ADR 0014) qui commit. Un nouveau commit-inside-service doit être justifié par le **même critère** : "le client ne doit pas pouvoir défaire cet effet de bord en provoquant l'exception qui le porte".
- **L'infra de test doit refléter la prod.** Tant que `tests/integration/conftest.py:_override_get_db` yield la session sans wrap commit/rollback, le bug se masque (le flush reste visible dans la session du test). L'override doit imiter `get_db` au moins pour les paths qui exercent commit-inside-service, sinon les tests deviennent un faux-positif structurel.
- **Pinning par session séparée.** Les tests qui assertent qu'un effet de bord critique a survécu à l'exception doivent **vérifier depuis une session distincte de celle du request**, sinon ils valident l'écriture mémoire SQLAlchemy plutôt que la persistance DB.
- **Futurs candidats à instruire au cas par cas** (pas un blanc-seing) : compteurs de rate-limit qui incrémentent puis raise 429, audit logs `security_event` sur tentatives de step-up 2FA échouées, tombstones d'invitations consommées concurremment. Chacun ré-évalué contre le critère ci-dessus avant d'adopter le pattern.
- **Pas d'usage du pattern dans le write upload handler** (ADR 0014) : la séquence des 10 étapes par mutation est explicitement **une transaction par mutation, commitée à l'étape 8**. L'append `sync_request_log` (étape 9) vit DANS cette transaction, **avant** le commit (delta D-B / S13.6, cf. ADR 0014) — write et ligne d'idempotence sont committés atomiquement, sans fenêtre crash-entre-commit-et-append (sinon double-write au replay). Le commit-inside-service de cet ADR ne s'applique pas ici : il n'y a aucun effet de bord de sécurité à committer *avant un `raise`* — une exception domaine rollback la mutation N et le dispatcher la mappe en `WriteResult.error` (récupérable) plutôt que de la propager.
