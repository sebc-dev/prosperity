# E08.5 — Réconciliation dépense confirmable ↔ consommation budget

> **Durée estimée** : 1-2 jours
> **Statut** : not started
> **Dépend de** : E08
> **Bloque** : E11 (le livrable overflow F10 exige qu'une transaction confirmée consomme un budget)
> **ADRs activés** : [0017](../adr/0017-confirmable-expense-vs-budget-consumption.md) (nouveau)
> **Suivi** : [#133](https://github.com/sebc-dev/prosperity/issues/133)

---

## Objectif

Lever la contradiction de modélisation documentée par l'ADR 0017 : la règle de confirmation E07 (`assert_expenses_categorized` refuse **toute** jambe `category_id IS NULL` d'une dépense non-transfert) et le modèle de consommation E08 (`compute_consumption` présuppose une **forme canonique à jambe compte `category_id = NULL`** + jambe catégorie) sont mutuellement exclusifs — **aucune transaction confirmable par le service ne peut faire monter une consommation budget**. Prouvé par `test_confirm_planned_categorised_200` (nette à 0) et `test_confirm_uncategorised_expense_422` (rejetée) ; toute la suite budget masque le mur en seedant la forme consommatrice en DB directe.

Option 1 de l'ADR retenue : **marqueur structurel de rôle de jambe sur `Split`**. La jambe `funding` (mouvement de compte) peut être non catégorisée ; seules les jambes `classification` exigent une catégorie. La forme canonique consommatrice devient alors confirmable via le flux réel.

Livrable agrégé : une dépense saisie via HTTP (`POST /accounts/{id}/transactions` → `POST /transactions/{id}/confirm`) en forme canonique (jambe `funding` compte + jambe `classification` catégorie) se confirme **et** fait monter la consommation du budget de la catégorie — sans seed DB.

## Critères d'acceptation

- [ ] Une dépense en forme canonique (jambe `funding` compte NULL + jambe `classification` catégorisée) se confirme via le service **et** fait monter la consommation du budget de sa catégorie
- [ ] Une jambe `classification` sans catégorie est toujours refusée à `confirmed` (`UncategorizedExpenseError`)
- [ ] Les scénarios threshold S08.3.4 sont pilotés via le flux confirm réel (plus de seed DB / plus de `_add_planned_trigger` net 0)
- [ ] Le parcours E2E `test_budget_lifecycle.py` passe (consommation produite par le flux HTTP, pas par seed)

## Notes pour l'implémenteur

- 🔒 **Changement de modèle pris au coût minimal.** La migration `0013` + le changement de sync rules + le back-fill arrivent **avant qu'aucune instance ne soit en prod** (roadmap `not started`). C'est l'argument central de l'option 1 vs l'option 2 (relâchement par convention sans colonne) dans l'ADR 0017.
- ⚠️ **E08 (budgets) n'a rien à changer** : son modèle de consommation (`_consumption_filters`, exclusion de la jambe `category_id IS NULL`) présuppose déjà la forme canonique B. C'est E07 qui doit la produire de façon confirmable — cet epic clôt l'écart en faveur du code budget déjà écrit.
- ⚠️ **`is_transfer` inchangé** : un transfert (≥ 2 comptes) reste exempté de la catégorisation indépendamment de `leg_role`. La nouvelle règle ne touche que les non-transferts.
- La forme canonique d'une dépense devient **B** (funding + classification) et non plus la forme « 2 jambes même catégorie » qu'imposaient les seuls tests confirmables — les factories convergent vers cette forme unique.
