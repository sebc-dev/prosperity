## 1. État des lieux & décisions structurantes

**Ce qui existe déjà** (à réutiliser tel quel) :
- `backend/modules/auth/service/jwt.py` : `issue_access_token`, `verify_access_token`, `InvalidTokenError`, `ExpiredTokenError` (HS256 pinné, vérifié)

- `backend/modules/auth/models.py` : `User`, `RefreshToken` avec `family_id` (indexé) + `parent_id` (self-FK, ON DELETE SET NULL)
- `tests/factories/sqlalchemy.py::UserFactory` (Argon2id via pwdlib)
- `tests/integration/conftest.py::db_session` (rollback par test, testcontainers Postgres)
- `httpx>=0.28` et `fastapi>=0.118` dans les deps

**Ce qui manque** (S02.4 doit construire) :
- Lifespan FastAPI gérant l'engine async sur `app.state` + dépendance `get_db`
- Routeur HTTP `modules/auth/transports/http.py` + sous-module `transports/`
- Schémas Pydantic `modules/auth/schemas.py`
- `rotate()` côté service (le schéma est prêt, la logique ne l'est pas)
- Détection de replay (invalidation de famille) avec re-revocation après INSERT (race fix)
- Isolation REPEATABLE READ sur l'engine (defense-in-depth contre la race replay-vs-rotate)
- `get_current_user` exposé via `public.py`
- Fixture httpx `async_client` avec override de `get_db` pointant sur `db_session` du test

**Décisions structurantes** :

| Décision | Choix | Justification |
|---|---|---|
| Dépendance `get_settings()` | `Depends(get_settings)` côté routes, paramètre explicite `settings: Settings` côté helpers service | Résout la dette S02.2/S02.3 (`cache_clear()` forcé en tests). |
| Mapping exceptions → HTTP | try/except inline dans la route, toutes les exceptions refresh-token → 401 uniforme | Plus simple, plus local, plus testable. Pas d'énumération d'état côté client. |
| Rotation atomique | `UPDATE … WHERE token_hash=? AND revoked_at IS NULL AND expires_at > now() RETURNING …` + re-revocation famille après INSERT | Replay detection via la même race que TOCTOU. La re-revocation post-INSERT ferme la fenêtre où T2 réviste la famille pendant que T1 insère son successeur. |
| Isolation Postgres | `isolation_level="REPEATABLE READ"` sur l'engine (defense-in-depth) | Ferme la fenêtre READ COMMITTED où l'UPDATE famille de T2 ne voit pas l'INSERT non-commité de T1. Erreurs `SerializationError` rares en pratique sur ces requêtes ciblées. |
| Engine lifecycle | Créé dans `app.lifespan`, stocké sur `app.state.engine`, ré-injecté via `get_db` | Évite la dette `lru_cache` (réinit forcée en tests). Pattern FastAPI canonique. |
| `device_label` source | `Request.headers["user-agent"]` côté route login, sanitization stricte (whitelist ASCII printable, tronqué à 120 chars) | Le client ne contrôle pas l'UA, on whitelist pour bloquer BiDi/ZWJ/homoglyphes. Champ jamais rendu au client (documenté). |
| Fixture httpx | `async_client` qui override `get_db` pour yielder le `db_session` du test (sans commit) | Standard FastAPI ; le rollback du `db_session` annule tout en fin de test. |
| Test TOCTOU | Service-layer avec mock `session.execute` simulant `rowcount=0` sur le loser | Impossible d'exposer une vraie race à travers le `db_session` (1 connexion, 1 transaction). Le mock prouve la logique de branchement déterministe. |

### 3.2 Engine async + dépendance `get_db` via lifespan

**Nouveau fichier** : `backend/shared/db.py`

```python
"""Async SQLAlchemy engine + FastAPI dependency.

The engine is created at app startup via `lifespan` and stored on
`app.state.engine`. Tests construct their own engine via the
`db_engine` fixture, no global cache to clear.

REPEATABLE READ isolation closes the replay-vs-rotate race window
(cf. plan §5.1) where an UPDATE in a concurrent transaction would
miss an INSERT made by a still-uncommitted sibling transaction.
"""
from __future__ import annotations
from collections.abc import AsyncIterator
from contextlib import asynccontextmanager
from fastapi import FastAPI, Request
from sqlalchemy.ext.asyncio import AsyncEngine, AsyncSession, async_sessionmaker, create_async_engine

from backend.config import Settings


def build_engine(settings: Settings) -> AsyncEngine:
    return create_async_engine(
        settings.database_url,
        future=True,
        isolation_level="REPEATABLE READ",
    )


@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncIterator[None]:
    from backend.config import get_settings
    engine = build_engine(get_settings())
    app.state.engine = engine
    app.state.sessionmaker = async_sessionmaker(engine, expire_on_commit=False)
    try:
        yield
    finally:
        await engine.dispose()


async def get_db(request: Request) -> AsyncIterator[AsyncSession]:
    sessionmaker = request.app.state.sessionmaker
    async with sessionmaker() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise
```

**Décisions** :
- Commit-on-success côté dépendance ⇒ les routes n'ont pas à se soucier du commit.
- Engine REPEATABLE READ defense-in-depth pour `rotate()` (§5.1). Les `SerializationError` éventuelles bubblent en 500 ; pas un problème pratique pour les requêtes ciblées du module auth.
- Plus de `lru_cache` ⇒ pas de dette `cache_clear()` en test. L'app entière est ré-instanciable.
- **Risque résiduel commit-avant-sérialisation** : si Pydantic lève à la sérialisation après `await session.commit()`, le client reçoit 500 mais le DB est muté. Acceptable pour S02.4 (réponses `TokenPair` simples, risque sérialisation ~0). Documenter dans la docstring.

**Update `backend/main.py`** :
```python
from backend.shared.db import lifespan
app = FastAPI(lifespan=lifespan, ...)
```

### 4.1 Schémas Pydantic + sanitization

**Nouveau fichier** : `backend/modules/auth/schemas.py`

```python
from typing import Literal
from pydantic import BaseModel, EmailStr, Field


class LoginRequest(BaseModel):
    email: EmailStr
    password: str = Field(min_length=1, max_length=1024)


class TokenPair(BaseModel):
    access_token: str
    refresh_token: str
    token_type: Literal["bearer"] = "bearer"


class RefreshRequest(BaseModel):
    refresh_token: str = Field(min_length=1, max_length=512)


class LogoutRequest(BaseModel):
    refresh_token: str = Field(min_length=1, max_length=512)


DEVICE_LABEL_MAX = 120
# Whitelist : ASCII printable seul (0x20–0x7E). Bloque BiDi controls (U+202E),
# zero-width joiners, homoglyphes Cyrillic, etc. Le champ n'est jamais rendu
# au client (usage interne/admin uniquement) — cf. ADR sur device tracking.
_DEVICE_LABEL_ALLOWED = frozenset(chr(c) for c in range(0x20, 0x7F))


def sanitize_device_label(raw: str | None) -> str | None:
    """Sanitize User-Agent header for storage as device_label.

    NEVER render the result back to clients without HTML-escaping; this
    function only blocks log/console injection vectors, not XSS.
    """
    if not raw:
        return None
    cleaned = "".join(c for c in raw if c in _DEVICE_LABEL_ALLOWED).strip()
    return cleaned[:DEVICE_LABEL_MAX] or None
```

### 4.2 Route login

```python
from functools import cache
from fastapi import APIRouter, Depends, HTTPException, Request
from pwdlib import PasswordHash
from sqlalchemy import func, select
from sqlalchemy.ext.asyncio import AsyncSession

from backend.config import Settings, get_settings
from backend.modules.auth.models import User
from backend.modules.auth.schemas import LoginRequest, TokenPair, sanitize_device_label
from backend.modules.auth.service.jwt import issue_access_token
from backend.modules.auth.service.refresh_tokens import issue as issue_refresh
from backend.shared.db import get_db

router = APIRouter(prefix="/auth", tags=["auth"])


@cache
def _password_hasher() -> PasswordHash:
    # Lazy-init : PasswordHash.recommended() instancie Argon2id (50-200ms).
    # Évite le coût à l'import du module.
    return PasswordHash.recommended()


@cache
def _dummy_hash() -> str:
    # Pré-calcul lazy d'un hash bidon pour égaliser le timing
    # quand l'email n'existe pas (anti-énumération).
    return _password_hasher().hash("dummy")


@router.post("/login", response_model=TokenPair, status_code=200)
async def login(
    body: LoginRequest,
    request: Request,
    session: AsyncSession = Depends(get_db),
    settings: Settings = Depends(get_settings),
) -> TokenPair:
    user = (
        await session.execute(
            select(User).where(func.lower(User.email) == body.email.lower())
        )
    ).scalar_one_or_none()
    if user is None or user.disabled_at is not None:
        # Toujours exécuter verify() pour égaliser le timing Argon2id.
        _password_hasher().verify("dummy", _dummy_hash())
        raise HTTPException(401, "Invalid credentials")
    if not _password_hasher().verify(body.password, user.password_hash):
        raise HTTPException(401, "Invalid credentials")
    device_label = sanitize_device_label(request.headers.get("user-agent"))
    access = issue_access_token(user.id, settings=settings)
    refresh = await issue_refresh(
        session, user.id, settings=settings, device_label=device_label
    )
    return TokenPair(access_token=access, refresh_token=refresh)
```

## 5. P02.4.2 — Rotation, replay, TOCTOU, routes refresh + logout

### 5.1 Service `rotate()` + détection replay + re-revocation post-INSERT

**Modifier** : `backend/modules/auth/service/refresh_tokens.py`

```python
from typing import cast
from sqlalchemy.engine import CursorResult

async def rotate(
    session: AsyncSession, raw_token: str, *, settings: Settings
) -> tuple[UUID, str]:
    """Atomically revoke `raw_token`, issue a new one in the same family.

    Returns (user_id, new_raw_token). Raises:
      - InvalidRefreshTokenError: token unknown
      - ExpiredRefreshTokenError: expires_at < now
      - RevokedRefreshTokenError: already revoked → triggers family-wide
        invalidation (replay detected) before raising.

    Atomicity: the UPDATE … RETURNING is single-statement atomic. The engine
    runs at REPEATABLE READ (cf. shared/db.py) which prevents a concurrent
    replay's family-revocation UPDATE from missing the new token inserted
    here in a still-uncommitted sibling transaction. As defense-in-depth,
    we re-issue the family-revocation predicate after the INSERT to detect
    a race where another transaction marked the family as compromised.
    """
    token_hash = hash_refresh_token(raw_token, settings=settings)
    now = datetime.now(tz=UTC)

    # Étape 1 — UPDATE atomique sur la ligne live ; gagne la course exactement
    # une fois grâce au prédicat `revoked_at IS NULL`.
    update_result = cast(
        CursorResult[tuple],
        await session.execute(
            update(RefreshToken)
            .where(
                RefreshToken.token_hash == token_hash,
                RefreshToken.revoked_at.is_(None),
                RefreshToken.expires_at > now,
            )
            .values(revoked_at=now)
            .returning(
                RefreshToken.id, RefreshToken.user_id, RefreshToken.family_id
            )
        ),
    )
    row = update_result.one_or_none()

    if row is None:
        # Distinguer inconnu / révoqué / expiré pour le logging et le replay.
        existing = (
            await session.execute(
                select(RefreshToken).where(RefreshToken.token_hash == token_hash)
            )
        ).scalar_one_or_none()
        if existing is None:
            raise InvalidRefreshTokenError("Refresh token is invalid")
        if existing.revoked_at is not None:
            # Replay détecté → invalider toute la famille.
            await session.execute(
                cast(
                    CursorResult[None],
                    update(RefreshToken)
                    .where(
                        RefreshToken.family_id == existing.family_id,
                        RefreshToken.revoked_at.is_(None),
                    )
                    .values(revoked_at=now),
                )
            )
            await session.flush()
            raise RevokedRefreshTokenError(
                "Refresh token replay detected; family invalidated"
            )
        raise ExpiredRefreshTokenError("Refresh token has expired")

    # Note: row est un Row SQLAlchemy ; accès par attributs nommés.
    parent_id = row.id
    user_id = row.user_id
    family_id = row.family_id

    # Étape 2 — émettre le successeur dans la même famille.
    new_raw = secrets.token_urlsafe(_TOKEN_ENTROPY_BYTES)
    record = RefreshToken(
        user_id=user_id,
        token_hash=hash_refresh_token(new_raw, settings=settings),
        issued_at=now,
        expires_at=now + timedelta(seconds=settings.refresh_token_ttl_seconds),
        family_id=family_id,
        parent_id=parent_id,
    )
    session.add(record)
    await session.flush()

    # Étape 3 — defense-in-depth : si la famille a été marquée compromise
    # entre l'UPDATE step 1 et notre INSERT (race REPEATABLE READ
    # théoriquement impossible mais on assure), révoquer notre propre
    # nouveau token et lever.
    family_status = (
        await session.execute(
            select(RefreshToken.id)
            .where(
                RefreshToken.family_id == family_id,
                RefreshToken.id != record.id,
                RefreshToken.revoked_at.isnot(None),
                RefreshToken.parent_id.is_(None),  # tokens marqués replay
            )
            .limit(1)
        )
    ).scalar_one_or_none()
    if family_status is not None:
        await session.execute(
            update(RefreshToken)
            .where(RefreshToken.id == record.id)
            .values(revoked_at=now)
        )
        await session.flush()
        raise RevokedRefreshTokenError("Family compromised during rotation")

    return user_id, new_raw
```

**Notes** :
- L'`UPDATE … RETURNING` règle TOCTOU **et** replay en un seul SQL atomique : deux refresh concurrents → un seul matche `revoked_at IS NULL`, l'autre tombe en `RevokedRefreshTokenError` et déclenche l'invalidation famille.
- Le `expires_at > now` dans le WHERE évite qu'un token expiré tombe par erreur dans la branche « revoked ».
- L'isolation REPEATABLE READ (§3.2) + l'étape 3 ferment la fenêtre où un replay concurrent rate l'INSERT d'un nouveau token.
- `row.id`, `row.user_id`, `row.family_id` : accès par attributs nommés (le `.t` du draft initial était un bug).
- `cast(CursorResult[...], ...)` requis car `AsyncSession.execute` est typé `Result` ; pattern déjà en place dans `refresh_tokens.py:142-156` (S02.3).
- `verify()` reste inchangé — la route `/auth/refresh` appellera `rotate()`, pas `verify()`. `verify()` reste utilisé en tests et pour d'éventuels futurs consommateurs read-only.

### 5.2 Route `POST /auth/refresh`

```python
import logging
logger = logging.getLogger(__name__)

@router.post("/refresh", response_model=TokenPair, status_code=200)
async def refresh(
    body: RefreshRequest,
    session: AsyncSession = Depends(get_db),
    settings: Settings = Depends(get_settings),
) -> TokenPair:
    try:
        user_id, new_refresh = await rotate(session, body.refresh_token, settings=settings)
    except InvalidRefreshTokenError as exc:
        # Hiérarchie : capture Invalid + Expired + Revoked → 401 uniforme.
        # Le `reason` ne sort jamais vers le client ; logs internes uniquement.
        logger.info("refresh_failed", extra={"reason": type(exc).__name__})
        raise HTTPException(401, "Invalid refresh token") from exc
    access = issue_access_token(user_id, settings=settings)
    return TokenPair(access_token=access, refresh_token=new_refresh)
```

### 5.3 Route `POST /auth/logout`

```python
@router.post("/logout", status_code=204)
async def logout(
    body: LogoutRequest,
    session: AsyncSession = Depends(get_db),
    settings: Settings = Depends(get_settings),
) -> Response:
    # Idempotent : revoke()==0 (inconnu ou déjà révoqué) renvoie quand même 204.
    # Pas de signal différentiel au client (anti-énumération).
    token_hash = hash_refresh_token(body.refresh_token, settings=settings)
    await revoke(session, token_hash)
    return Response(status_code=204)
```

## 6. P02.4.3 — `get_current_user` + tests route protégée

### 6.1 Dépendance FastAPI

**Nouveau fichier** : `backend/modules/auth/transports/dependencies.py`

```python
from fastapi import Depends, HTTPException
from fastapi.security import HTTPAuthorizationCredentials, HTTPBearer
from sqlalchemy.ext.asyncio import AsyncSession

from backend.config import Settings, get_settings
from backend.modules.auth.models import User
from backend.modules.auth.service.jwt import InvalidTokenError, verify_access_token
from backend.shared.db import get_db

_bearer = HTTPBearer(auto_error=False)


async def get_current_user(
    credentials: HTTPAuthorizationCredentials | None = Depends(_bearer),
    session: AsyncSession = Depends(get_db),
    settings: Settings = Depends(get_settings),
) -> User:
    if credentials is None or credentials.scheme.lower() != "bearer":
        raise HTTPException(401, "Missing or invalid Authorization header")
    try:
        user_id = verify_access_token(credentials.credentials, settings=settings)
    except InvalidTokenError as exc:  # capture ExpiredTokenError (sous-classe) aussi
        raise HTTPException(401, "Invalid or expired access token") from exc
    user = await session.get(User, user_id)
    if user is None or user.disabled_at is not None:
        raise HTTPException(401, "User no longer exists or is disabled")
    return user
```

## 8. Hors scope (reporté à S02.5 / E16 infra)

Conformément au comment de l'issue et aux décisions de revue :
- **Cap par user + purge des refresh tokens expirés** → cron infra (E16)
- **Atomicité `revoke()` strict en concurrence** → même remède que TOCTOU, mais pas critique tant que logout reste idempotent
- **Test de concurrence Postgres réel** (2 connexions distinctes avec commits manuels, pour valider expérimentalement le claim atomicité du `UPDATE … RETURNING` au niveau base) → test e2e dédié hors S02.4
- **Rate-limiting `/auth/login`** → S02.5 ou couche infra (proxy/middleware). Une route login non-protégée est acceptable pour S02.4 mais doit être adressée avant prod.
- **Account lockout après N tentatives échouées** → S02.5
- **CSRF** : N/A pour S02.4. Confirmé : le refresh token reste dans le body JSON, **jamais en cookie** pour toute la vie d'E02. Si E03+ introduit un cookie, CSRF protection devra être ajoutée.
